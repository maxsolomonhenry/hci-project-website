II. Design Evolution



From the two low fidelity prototypes we explored, we decided to create an audio engine that spatialized based on a flat, "forward-to-backward" orientation rather than a newspaper-style, "up-and-down" orientation typically associated with websites. From here we gathered additional observations from the low-fidelity prototype recordings, classmates’ suggestions, and our meeting with T.A. Juliette. We decided to focus on the following four areas to improve in our computer prototype.

1. The first thing that improved was the particular sounds on our engine. One of the participants in our study mentioned that the sounds were "annoying," which could decrease the long-term usability of the product. We would like users to feel comfortable staying for a long time on certain webpages, if they so desire. Rather than sinusoidal beeps and boops, the new system features pleasant, musical sounds with reverb, that easily guide users to find the buttons.

2. Next we had to address a fundamental issue with the way users spatialize sound internally. One of the participants noted that it took a lot of time for them to understand "why we move the speaker in a reverse position of the cursor". It became clear to us that users have to be made to think of the system not as a auralizin a flat webpage, but as creating a 3D environment. When we move our heads to the left, sound moves (relatively) to the right. This notion is a part of our everyday experience, and it was important to communicate to the user that they need to think of the system on this, somewhat different way. As a result, we provide a detailed onboarding tutorial, and emphasize this notion in the user guide with a step-to-step instruction and a detailed explanation of the concepts of moving. Please review our User manual and installation guide for a detailed explanation.

3. It occured to us that the system had little feedback, aside from orientation. Juliette noted that she could not figure out if she clicked the button or not with her eyes closed. She suggested that we should have an indication of whether the user clicked the button or not to increase usability and user experiences. Now there is a sound that indicates that the user has clicked, and a new page has loaded (it's a pleasant "ding" sound).

4. As mentioned earlier, we opted to use a so-called "straight-forward" rather than an up-and-down newspaper model for the audio engine. In our low-fidelity prototype, we tested straight-forward and newspaper-style models. Then we got the result that the newspaper could be the best solution. However, when we started the computer prototype, we found that it is not feasible to build our website in the newspaper way since it will be hard for the user to find whether the sound is lower or upper and is hard to implement. So we built our website with the straight-forward model, the sounds are moving in front and behind instead of up and down. Now it is feasible to test all functions.

IV. User population

Our project proposal establishes our target population is those who have been non-sighted from birth; specifically, those who (1) have no visual preconceptions about what the internet looks like or what a website looks like , and (2) are accustomed to physically navigating their world through aural cues. We suspect this population has a superior capacity to build an internal model of an environment based on its reverberant characteristics. That said, we acknowledge that it is difficult to find experts from our target population that are within this class. If you are not visually impaired, we request that you close your eyes when evaluating the system on the demo sight (after the tutorial).



V. Usability goals

1. Learnability: The user should be able to learn the sound mapping quickly.

We want our users to quickly understand how moving the cursor affects the sounds they hear. Less than 2 minutes on the first page, the user should have found the button and clicked on it.

2. Ease of use: The user should be able to identify the position of multiple objects on the screen and determine how they are placed in reference to one another.

The system should allow users to feel that the objects have their own space and that the cursor helps them move towards the object. The user should not have to be reminded of how to navigate the system more than twice. The user should be able to perform every task without accessing the User Manual more than twice.

3. Efficiency: The system should not be stressful or cause any sort of sound anxiety to the user. Instead, it should feel helpful and inviting.

The system should feel inviting, comfortable and help the users orient themselves by sound cues. After three minutes of use, the user should not demonstrate any visible signs of frustration, stress, anxiety, or confusion. Visible signs to be considered are: facial expressions, physical movements (hands and legs moving uncomfortably, tense), comments, or extensive mouse movements.

4. Effectiveness: The system should permit the user to find objects without thinking too much.

Users should be able to navigate into pages and directly follow the sound cues towards an object of interest. The auditory scene should be inviting and pleasant to the user, who will become familiar with the sound environment and find objects fast. The user should not spend more than 1 minute on a page, except for the first page, where they should spend 2 minutes at maximum.

5. Satisfaction: Users should feel like their browsing experience has become better and faster. They should feel like they know where everything is on the website.

This project aims to help visually impaired users with the issue of orientation when navigating websites. By the end, users should be confident of where buttons are. Users should be able to get to the end of the prototype in under 6 minutes.



VI. Usability test procedure


Click the follow link to download our test plan:
Usability Test Plan

How many examiners are required?

Because we want to gather different comments and results from different people, 2 to 3 examiners are required during our test. There are only two roles in our usability test: one is the user and another one is the examiner (observer). We need 3 to 5 users (participants), and there should be 2 to 3 examiners (observers) present.

What equipment will the examiners need?

For the technology part, our computer prototype is built on a website, so they need access to a computer and the internet. Our system is designed for visually impaired people, so the examiners need something to limit their eyesight like blindfolds. They also need headphones for our audio display. For the documentation part, they need to have the Usability Test Plan, User Manual and Installation Guide, and Usability Goals in hand.

How should your prototype be handled? (manual/documentation)

It should be handled as a website being accessed from a computer.

How will you instruct your examiners to proceed?

The user needs to read the user manual and installation guide before they do the test. Examiners need to have it at hand and remind the user what to do next when he/she is lost (but not how to do it). Both examiners and users should have usability goals in mind and read the test plan before the test starts. Before and after the test, they should read about how to report and how to evaluate twice.

How should the examiners treat the test subjects, what should they tell them?

Examiners and users should treat the test subjects with kindness and patience. They should read the user manual and installation guide carefully before the test and think about their tasks and what we want them to think during the test. Examiners should avoid telling the user how to do the tasks, but remind him/her of what he/she needs to do when he/she is lost.

What should the examiners avoid doing?

Examiners should avoid giving the user directions, personal opinions, feedback (verbal), interrupts, or making noises. Examiners and users should avoid conducting the test in a noisy environment because we have an audio display that plays an important role in our system.

What should the examiners avoid telling the test subjects?

Examiners should avoid giving feedback on how well the user is performing. Also, they need to avoid telling the user how to do a task.

What/How/When/Where should the examiners measure?

What & How & When: Examiners should present the user with the prototype and measure the time he/she spends on completing each task. They should also count the number of mistakes or pauses the user makes when completing each task.
Where: This is very important! Examiners need to find a quiet environment and make sure the user can hear our prototype’s sound clearly with a pair of headphones on.

VIII. Usability Evaluation

Please use the Heuristic Evaluation Sheet for the usability evaluation. Based on Nielsen’s heuristic evaluation, these are the specific topic to be assessed:

1. Visibility of system status: The design should always keep users informed about where they are and what is going on through frequent feedback.

Does the system provide good aural feedback according to the user's position and actions? Are the sounds comprehensible, distinguishable, and indicative of the user's actions?

2. Match between the system and the real world: The design should use words, phrases, sounds, and concepts familiar to the user while following real-world conventions and making information appear in a logical order.

Do the sounds properly indicate the position of buttons? Can the users comprehend the button-sound mapping? Are the users able to navigate the website with knowledge familiar to them?

3. User control and freedom: Users need to have an "emergency exit" in case they perform an unwanted action. This exit must not involve multiple steps.

Does the system allow the user to go back to the previous page? Does the system allow users to consult an instructions page at any point? Does the system have any page where the user could be stuck in without knowing what to do?

4. Consistency and standards: Users should not have to know whether different sounds indicate the same thing. The system should have a clear sound-object mapping and follow conventions.

Does the user demonstrate trouble distinguishing the different sounds of the system? Is the user able to understand what each sound indicates?

5. Error prevention: The best designs carefully prevent problems from occurring,either by eliminating error-prone conditions, or by checking for them and presenting users with confirmation options before they commit to the action.

Does the system contain sources for error? Does the user give feedback to indicate actions are about to be taken? Does the system guide the user towards making a correct decision?

6. Recognition rather than recall: User's memory load should be minimized by making elements, sounds, and options recognizable. The user should not have to remember information from one part of the interface to another. Information should be easily retrievable.

Does the system contain familiar and recognizable sounds? Does the user get lost every time they access a new page? Are the users able to recognize where they are at any given time?

7. Flexibility and efficiency of use: Allow users to tailor frequent actions, giving the options of shortcuts that may speed up the interaction process. The design should cater to both inexperienced and experienced users.

Does the system allow for experienced users to go straight to the objects of interest? Is the sound feedback fast enough for experienced users? Does the sound slow down experienced users?

8. Aesthetic and minimalist design: Every unit of information in an interface should not be irrelevant or rarely needed.

Is every sound useful? Does the system contain elements that are rarely used, or not used at all?

9. Help and documentation: For some complex systems, it may be necessary to provide documentation to help users understand how to complete their tasks.

Does the system have sufficient documentation to help users? Is the documentation helpful and easily accessible? Does it contain a tutorial? Does it have a user manual with precise instructions on how to operate the system?
